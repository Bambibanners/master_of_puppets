Write `docs/docs/developer/setup-deployment.md` — 200+ lines, fresh from scratch, operator-first structure. Do not port content from legacy files.

**Structure (in this order):**

**1. Quick Start** (top of document, before any other section)
A clearly labelled 5-command block that gets anyone to a running production stack without reading further. Use a `!!! tip "Get running in 60 seconds"` admonition box containing the commands:
```bash
git clone <repo>
cd puppeteer
cp .env.example .env   # fill in SECRET_KEY, ENCRYPTION_KEY, ADMIN_PASSWORD, API_KEY (see env vars below)
docker compose -f compose.server.yaml up -d
# Dashboard available at https://localhost (Caddy self-signed) or your CF tunnel domain
```
Note that the stack takes ~15 seconds to become healthy. Point to the "Environment Variables" section for the values needed before `docker compose up`.

**2. Prerequisites**
- Docker 24+ and Docker Compose v2
- (For local dev only) Python 3.12, Node.js 18+

**3. Production Deployment (Docker Compose)**
Section covering the full Docker Compose path:
- Services table: list all 10 services from compose.server.yaml (db, cert-manager, agent, model, dashboard, docs, registry, pypi, mirror, tunnel) with image, port, and purpose
- How to bring up: `docker compose -f puppeteer/compose.server.yaml up -d`
- How to rebuild after code changes: `docker compose -f puppeteer/compose.server.yaml build agent && docker compose -f puppeteer/compose.server.yaml up -d --no-build agent`
- Logs: `docker compose -f puppeteer/compose.server.yaml logs -f agent`
- Note: DO NOT reference `mkdocs serve` for the docs container — production uses nginx (mention this explicitly with a note admonition)

**4. Environment Variables**
Full table documenting ALL required env vars. Mark required vs optional clearly. Include placeholder example values and generation commands:

| Variable | Required | Purpose | Example / How to Generate |
|---|---|---|---|
| `API_KEY` | **REQUIRED** | Legacy shared key; security.py exits at startup if missing | `openssl rand -hex 32` |
| `ENCRYPTION_KEY` | Required (unstable if absent) | Fernet key for job secrets at rest | `python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"` |
| `SECRET_KEY` | Recommended | JWT signing key | `openssl rand -hex 32` |
| `ADMIN_PASSWORD` | Recommended | Initial admin password | Any secure password |
| `DATABASE_URL` | Optional | DB connection string (defaults to SQLite) | `postgresql+asyncpg://user:pass@host/db` |
| `AGENT_URL` | Required for nodes | URL nodes use to reach agent_service | `https://192.168.1.10:8001` |
| `CLOUDFLARE_TUNNEL_TOKEN` | CF tunnel only | Cloudflare tunnel credentials | From CF Dashboard |
| `DUCKDNS_TOKEN` / `DUCKDNS_DOMAIN` | DuckDNS only | Dynamic DNS for cert-manager | From DuckDNS |

Include a `!!! warning` admonition: "These docs are behind Cloudflare Access. Even so, never commit real keys to git. Use `.env` files (gitignored) or a secrets manager."

**5. Local Development**
The local dev section comes AFTER production — most readers are deploying, not hacking.

Subsections:

5a. Backend (FastAPI)
```bash
cd puppeteer
pip install -r requirements.txt
pip install aiosqlite  # required for SQLite local dev — NOT in requirements.txt
```
Required env vars for local dev (minimum before `python -m agent_service.main` starts):
```bash
export API_KEY=dev-key
export ENCRYPTION_KEY=$(python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())")
export SECRET_KEY=dev-secret-key
```
Include `!!! warning "aiosqlite required for SQLite"` — explain that security.py calls `sys.exit(1)` at import time if `API_KEY` is not set; must set env vars before launching.

Then: `python -m agent_service.main` starts on https://localhost:8001 (self-signed TLS).

5b. Frontend (React / Vite)
```bash
cd puppeteer/dashboard
npm install
npm run dev
```
Dev server runs at http://localhost:5173 and proxies `/api` to `https://localhost:8001` per `.env.development`. Backend must be running for API calls to work.

5c. Running Tests
```bash
# Backend
cd puppeteer && pytest

# Frontend
cd puppeteer/dashboard && npm run test
```

**6. TLS Bootstrap & Node Enrollment**
Brief section (not a deep dive — link to the future Security guide):
- After first login, find the JOIN_TOKEN in the Admin panel of the dashboard
- Set `JOIN_TOKEN` env var on the puppet node container
- Set `AGENT_URL` env var to the puppeteer's accessible address (e.g., `https://192.168.1.10:8001`)
- Node will auto-enroll on first startup and appear in the Nodes view
- Include: `!!! note "Deep cert rotation and mTLS details"` — link placeholder text "See the Security Guide (coming in a later documentation phase)" since Phase 24 is where that lives

**7. Upgrading**
Short section:
- Pull latest: `git pull && docker compose -f puppeteer/compose.server.yaml build && docker compose -f puppeteer/compose.server.yaml up -d`
- DB migrations: New deployments handled automatically by `create_all`. For existing deployments, check `puppeteer/migration_v*.sql` files for any new columns needed. See the Contributing guide for the migration pattern.
