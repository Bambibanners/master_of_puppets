**Part A: Create `docs/docs/developer/contributing.md`** — 150+ lines, fresh from scratch.

Required sections (in this order):

**1. Overview**
Short intro: "This guide covers how to contribute to Master of Puppets — from running tests locally to the PR bar and database migration conventions."

**2. Code Style**
Python formatting and linting tools. Include setup instructions:
```bash
pip install black ruff
```
Subsections:
- **Black (formatter):** `black puppeteer/` — formats Python code to 88 chars line length, Python 3.12 target. Configuration in `puppeteer/pyproject.toml`.
- **Ruff (linter):** `ruff check puppeteer/` — replaces flake8/isort/pyupgrade in a single fast tool. Rules: E (pycodestyle errors), F (pyflakes), I (isort), W (warnings); E501 (line-too-long) ignored because Black handles length.
- **Frontend (ESLint):** `cd puppeteer/dashboard && npm run lint` — already configured; must pass before PR.
- Note: Do NOT run `black .` or `ruff --fix .` on existing code in the same PR as your feature changes. Style-only reformatting should be a separate commit/PR to keep diffs reviewable.

**3. Running Tests**
Explicit commands with the exact cwd. Both must be green before a PR can merge.

```bash
# Backend (pytest)
cd puppeteer
pytest

# Frontend (vitest)
cd puppeteer/dashboard
npm run test
```

Include: `!!! tip "Quick run (fail-fast)"` showing `pytest tests/ -x -q` for the backend during development.

Mention: There is no numeric coverage threshold. Green = pass. The bar is: all existing tests continue to pass and new code has tests for its core behavior.

**4. PR Requirements**
What a passing PR requires (explicit checklist format):

- [ ] `cd puppeteer && pytest` passes with no failures
- [ ] `cd puppeteer/dashboard && npm run test` passes with no failures
- [ ] `cd puppeteer/dashboard && npm run lint` passes with no errors
- [ ] New Python code follows Black formatting (`black --check puppeteer/`)
- [ ] New Python code passes Ruff lint (`ruff check puppeteer/`)
- [ ] New API routes include a `tags=` parameter (17 tag groups established — check existing routes for examples)
- [ ] New DB columns include a migration SQL file (see Database Migrations section)

**5. Database Migrations**

Start with a `!!! warning "Major contributor gotcha — read this before adding DB columns"` admonition.

Content:
- No Alembic in this project. `Base.metadata.create_all` runs at startup and creates new tables — it will NOT alter columns in existing tables.
- Fresh deployments: fully covered by `create_all` — no migration steps needed.
- Existing deployments: if you add a column to an existing DB model, you MUST also create a `puppeteer/migration_vN.sql` file with the necessary `ALTER TABLE` statement.

Migration file format:
```sql
-- migration_v32.sql (increment from current highest: migration_v31.sql)
ALTER TABLE my_table ADD COLUMN IF NOT EXISTS new_column TEXT;
-- Use IF NOT EXISTS to make the migration idempotent (safe to run twice)
```

Rules:
- Use `IF NOT EXISTS` always — makes migrations idempotent
- SQLite uses `ALTER TABLE x ADD COLUMN y type` (no IF NOT EXISTS support — add a comment noting it's for Postgres)
- Never drop columns in a migration — destructive; coordinate separately
- File naming: sequential integer suffix, e.g., `migration_v32.sql`

**6. Code Structure**
Brief orientation to the codebase layout. Reference CLAUDE.md for the full table. Highlight key patterns:
- All FastAPI route handlers live in `puppeteer/agent_service/main.py` — do not split routes across multiple files
- Business logic goes in `puppeteer/agent_service/services/` — one service per domain
- Pydantic models for request/response in `puppeteer/agent_service/models.py`
- SQLAlchemy ORM models in `puppeteer/agent_service/db.py`
- All authenticated API calls from the frontend go through `authenticatedFetch()` in `dashboard/src/auth.ts`

**7. WebSocket and Real-Time Updates**
Short section: WebSocket endpoint at `/ws?token=<jwt>` pushes job/node updates to the dashboard. When adding events, emit via the WebSocket manager in `main.py`. Client-side hook is `dashboard/src/hooks/useWebSocket.ts`.

**8. Getting Help**
Point to the architecture guide (`developer/architecture.md`) for system understanding and the API reference (`/docs/api-reference/`) for endpoint schemas.
