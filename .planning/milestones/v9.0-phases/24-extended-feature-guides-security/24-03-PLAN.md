### Admin behaviour
Admin bypasses ALL permission checks — no DB lookup. Admin is the only role that can manage users, roles, and permissions.

### Permission grant/revoke API (main.py)
- `POST /admin/roles/{role}/permissions` — body: `{"permission": "jobs:read"}` — requires `users:write`
- `DELETE /admin/roles/{role}/permissions/{permission}` — requires `users:write`

### ServicePrincipal model (db.py)
- `client_id` — UUID used as identifier in tokens
- `client_secret_hash` — bcrypt hash of generated secret
- `role` — defaults to `"operator"`, can be any valid role
- `expires_at` — nullable datetime; SP tokens rejected if past this date
- `is_active` — boolean flag; SP tokens rejected if false
- JWT: `"type": "service_principal"`, `"sp_id"` in payload

### _SPUserProxy pattern (main.py)
Service principals masquerade as users for permission checks. Audit log shows `sp:<name>` as the username.

### User signing keys and API keys (db.py)
- `UserSigningKey`: per-user Ed25519 keys, managed in **My Account**
- `UserApiKey`: prefix stored in `key_prefix`; auth via `mop_` prefix detection

### Permission cache (main.py _perm_cache)
Permissions are cached per role. Cache invalidated by `_invalidate_perm_cache(role)` on permission change. Effect is immediate for new requests; existing cached tokens are not affected (token_version handles user-level invalidation separately).

**H1: Role-Based Access Control**
One-sentence intro: Master of Puppets uses three roles — `admin`, `operator`, and `viewer` — with a permission model that can be customised through the dashboard.

**H2: Roles Overview**
Brief description of each role:
- **admin**: Full unrestricted access. Bypasses all permission checks. Use only for initial setup and user provisioning.
- **operator**: Day-to-day platform access — can manage jobs, nodes, Foundry, and signatures. Cannot manage users or system config.
- **viewer**: Read-only access — can view jobs, nodes, Foundry content, and signatures but cannot make changes.

Add cross-link: "For the complete permission matrix showing which permissions are assigned to each role by default, see the [RBAC Permission Reference](rbac-reference.md)."

**H2: Managing Users**
Walk through the **Users** section of the dashboard.

Creating a user:
1. Navigate to **Users** in the sidebar
2. Click **Create User** and fill in username, email, and initial password
3. Select the role: `admin`, `operator`, or `viewer`
4. Click **Save** — the user can now log in

Changing a user's role:
- Click the role badge next to a username → select new role → confirm

Force password change on next login:
- Toggle **Force Password Change** on the user row — user will be prompted to change password on next login; all existing sessions remain valid until they next authenticate

Deleting a user:
- Click **Delete** on the user row
- Add `!!! warning`: "Deleting a user is permanent and cannot be undone. Any API keys or signing keys associated with that user are also removed."

Admin password reset:
- Click **Reset Password** on a user row to generate a new password — this immediately invalidates all active sessions for that user (token version is incremented).

**H2: Customising Role Permissions**
Permissions can be granted or revoked on a per-role basis through **Admin** → **Role Permissions**.

Add `!!! warning`: "Permission changes affect all users with that role immediately. Adding `users:write` to the `operator` role gives every operator the ability to manage users — use with care."

UI walkthrough:
1. Navigate to **Admin** → **Role Permissions**
2. Select a role tab (operator / viewer)
3. Click a permission chip to grant it; click again to revoke
4. Changes take effect immediately for new requests

Note: The `admin` role's permissions cannot be modified — admin always has full access.

**H2: Service Principals**
Service principals are machine identities — they are not human users and do not use the login flow. They are designed for automation and CI/CD pipelines.

"A service principal has a `client_id` and a `client_secret`, and is assigned a role just like a user. Actions performed by a service principal are attributed in the audit log as `sp:<name>`."

Creating a service principal:
1. Navigate to **Admin** → **Service Principals**
2. Click **Create Service Principal**
3. Enter a name (e.g. `ci-pipeline`) and select a role
4. Optionally set an expiry date
5. Click **Create** — the `client_secret` is shown **once** and cannot be retrieved again

Add `!!! danger`: "Copy the client secret immediately after creation. It is shown only once and cannot be retrieved. If lost, rotate the secret using the **Rotate Secret** button."

Rotating a service principal secret:
- Click **Rotate Secret** — the new secret is returned once; the old secret is immediately invalid

Deactivating a service principal:
- Toggle **Active** to false — the SP's tokens are immediately rejected without deleting it

Cross-link: "To get a JWT for a service principal and use it in CI/CD pipelines, see the [OAuth & Authentication guide](oauth.md)."

**H2: API Keys**
API keys are long-lived tokens for a specific user, managed in **My Account**. They start with `mop_` and can be used in automation in place of JWT tokens. See the [OAuth & Authentication guide](oauth.md) for usage details.
