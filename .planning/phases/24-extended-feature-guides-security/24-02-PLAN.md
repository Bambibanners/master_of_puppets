### Job Scheduling ŌĆö ScheduledJob model fields (db.py)
Targeting:
- `target_node_id` ŌĆö specific node UUID (explicit targeting)

### Job Scheduling ŌĆö status lifecycle (scheduler_service.py)
- `DRAFT` ŌĆö created via mop-push, not yet promoted
- `ACTIVE` ŌĆö running, will fire on schedule
- `DEPRECATED` ŌĆö scheduler skips firing, logs `job:deprecated_skip`
- `REVOKED` ŌĆö scheduler skips firing, logs `job:revoked_skip`
- Cron overlap guard: if previous instance still PENDING/ASSIGNED/RETRYING, new fire skipped, logs `job:cron_skip`

### Job Scheduling ŌĆö signature requirement
Every ScheduledJob must have `signature_id` and `signature_payload`. Scheduler validates signature via SignatureService before creating execution job.

### OAuth ŌĆö device flow endpoints (main.py)
- `POST /auth/device` ŌĆö initiates, returns `device_code`, `user_code`, `verification_uri`, `expires_in=300`, `interval=5`
- `POST /auth/device/token` ŌĆö polls with `device_code`; returns `slow_down` / `authorization_pending` / JWT on success
- `GET /auth/device/approve` ŌĆö inline HTML approval page
- `POST /auth/device/approve` ŌĆö admin approves the flow
- Device codes stored in-memory (`_device_codes` dict) ŌĆö NOT persisted across restarts

### OAuth ŌĆö token facts (auth.py)
- Algorithm: HS256
- Default expiry: `ACCESS_TOKEN_EXPIRE_MINUTES = 1440` (24 hours)
- JWT payload: `sub` (username), `role`, `tv` (token_version), `exp`
- Device flow tokens include `"type": "device_flow"`
- Service principal tokens include `"type": "service_principal"`, `"sp_id"`

Replace the stub with the full OAuth & Authentication guide. Structure:

**H1: OAuth & Authentication**
One-sentence intro: Master of Puppets uses JWT-based authentication with RFC 8628 device flow for CLI tools and API keys for automation.

**H2: Authentication Methods**
Brief overview table of the three methods: password login (dashboard), device flow (CLI/mop-push), API key (automation/CI).

**H2: Device Flow (OAuth 2.0)**
Conceptual explanation: "The OAuth 2.0 Device Authorization Grant (RFC 8628) allows tools without a browser ŌĆö like `mop-push` ŌĆö to authenticate without a redirect URI. The flow works by having the user approve the request in a browser session while the tool polls for the result."

Why MoP uses device flow: CLI tools cannot host a redirect URI. The device flow separates the user approval from the tool waiting for a token.

Flow steps (conceptual, no detailed commands ŌĆö those are in the mop-push guide):
1. CLI requests a `device_code` and `user_code` from `POST /auth/device`
2. User visits the `verification_uri` in a browser and enters the `user_code`
3. An admin (or the same user) approves the request at the approval page
4. CLI polls `POST /auth/device/token` every 5 seconds
5. On approval, the token endpoint returns a JWT

Constants: authorization window = 5 minutes, poll interval = 5 seconds.

Add `!!! warning`: "Device authorization codes are stored in memory and are not persisted across server restarts. If the server restarts during the 5-minute authorization window, the login flow must be restarted."

Cross-link: "For step-by-step mop-push login instructions, see the [mop-push CLI guide](mop-push.md)."

**H2: Token Lifecycle**
- Default expiry: 24 hours from issuance (JWT `exp` claim)
- Token fields: `sub` (username), `role`, `tv` (token_version), `exp`
- Service principal tokens include `type: "service_principal"` and `sp_id`
- Device flow tokens include `type: "device_flow"`

**Forced invalidation:**
"When a user's password is changed ŌĆö either by the user via **My Account** or by an admin ŌĆö all existing tokens for that user are immediately rejected. The JWT includes a `tv` (token version) field; any token with a `tv` that does not match the user's current version is invalid. There is no need to wait for token expiry."

Admin password reset: `POST /admin/users/<USERNAME>/reset-password` invalidates all active sessions for that user.

**H2: API Keys**
API keys are long-lived bearer tokens prefixed with `mop_`. They are created in the **My Account** section of the dashboard.

Usage (with `<PLACEHOLDER>` syntax):
```bash
curl -H "Authorization: Bearer mop_<YOUR_API_KEY>" \
  https://<YOUR_HOST>/api/nodes
```

Add `!!! warning`: "API keys grant the same access as the owning user's role. The `permissions` field is reserved for future per-key permission scoping ŌĆö per-key permission restriction is not currently enforced."

Revoking: API keys can be revoked in **My Account** ŌåÆ **API Keys** ŌåÆ **Revoke**.

**H2: Service Principal Tokens**
Service principals are machine identities (not human users) ŌĆö documented fully in the [RBAC guide](rbac.md). To get a token:

```bash
curl -X POST \
  -u "<SP_CLIENT_ID>:<SP_CLIENT_SECRET>" \
  https://<YOUR_HOST>/auth/service-principals/<SP_ID>/token
```

Secret rotation: `POST /auth/service-principals/<SP_ID>/rotate`. The new secret is returned once and cannot be retrieved again. Old secret is immediately invalid.

**H2: CI/CD Integration**
Practical section for pipeline automation. Two approaches:

*Approach A: API key (simpler)*
1. Create an API key in the dashboard (**My Account** ŌåÆ **API Keys** ŌåÆ **Create**)
2. Store as a CI secret (e.g. `MOP_API_KEY` in GitHub Actions secrets)
3. Use in pipeline:
```bash
curl -H "Authorization: Bearer mop_<MOP_API_KEY>" \
  https://<YOUR_HOST>/api/jobs/push \
  -d @job_payload.json
```

Or with mop-push:
```bash
MOP_API_KEY=mop_<MOP_API_KEY> mop-push push --script my_job.py
```

*Approach B: Service principal (recommended for teams)*
1. Create a service principal (**Admin** ŌåÆ **Service Principals** ŌåÆ **Create**)
2. Set role to `operator` (not `admin`) per least-privilege principle
3. Store `client_id` and `client_secret` as CI secrets
4. Request a token at pipeline start, use for the run duration

Add `!!! tip`: "Service principals support expiry dates (`expires_at`). Set a rotation schedule in your CI system to refresh the secret before it expires."
