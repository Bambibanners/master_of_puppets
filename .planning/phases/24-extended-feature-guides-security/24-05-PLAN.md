**H2: Principle of Least Privilege**

"Assign the minimum role needed for each user's actual job function. The default roles are designed for this:"

| User type | Recommended role | Rationale |
|-----------|-----------------|-----------|
| Monitoring-only operator | `viewer` | Read-only; cannot submit or modify anything |
| Day-to-day job operator | `operator` | Can manage jobs and nodes; cannot touch users or system config |
| Platform administrator | `admin` | Use only for setup and user provisioning; not for day-to-day work |
| CI/CD pipeline | Service principal (`operator`) | Machine identity; no login; expiry date enforced |

Add `!!! warning`: "Do not use the `admin` account for day-to-day operations. Admin bypasses all permission checks, which means admin actions provide no RBAC signal in the audit log — every admin action appears as 'permitted' regardless of what it is. Use a named operator account for routine tasks."

**H2: Service Principal Hygiene**

Service principals are the recommended machine identity for all automation. Hardening checklist:

- [ ] Set an expiry date (`expires_at`) on every service principal — do not create non-expiring SPs
- [ ] Assign the `operator` role, never `admin`, unless admin-level access is genuinely required
- [ ] Use one service principal per automation context (one for CI, one for monitoring scripts, etc.) — avoid shared SPs
- [ ] Rotate secrets on a schedule (quarterly or after any potential leak)
- [ ] Deactivate (`is_active = false`) rather than deleting a compromised SP — deletion removes the audit trail

Add `!!! danger`: "A service principal with the `admin` role has full unrestricted access. Treat its client secret with the same care as the admin password."

**H2: Auditing Permissions**

"Review current role permissions regularly to catch permission creep — permissions that were granted temporarily and never revoked."

Steps:
1. Navigate to **Admin** → **Role Permissions**
2. Review the current permission set for each non-admin role
3. Remove any permissions that are no longer required

Via API (for automation):
```bash
# List current permissions for the operator role
curl -H "Authorization: Bearer <TOKEN>" \
  https://<HOST>/admin/roles/operator/permissions

# Revoke a permission
curl -X DELETE \
  -H "Authorization: Bearer <TOKEN>" \
  https://<HOST>/admin/roles/operator/permissions/webhooks:write
```

Note: permission changes are logged as `permission:grant` and `permission:revoke` in the audit log.

**H2: Reviewing the Audit Log for Access Patterns**

"The audit log records all permission changes. For a periodic access review:"

- Search for `permission:grant` events in the last 90 days — review any unexpected grants
- Search for admin actions (`username = <admin username>`) — admin bypasses RBAC; any admin action should be intentional
- Look for `sp:*` actions from service principals that have been deactivated

Cross-link: "For query patterns and the complete event inventory, see [Audit Log](audit-log.md)."

**H2: Hardening the Default Configuration**

Optional hardening steps beyond the defaults:

1. **Restrict viewer permissions further**: Remove `signatures:read` from `viewer` if signature visibility is sensitive in your environment
2. **Remove `tokens:write` from `operator`**: If operators should not be able to approve device flow requests, remove this permission
3. **Add `users:write` only when needed**: Granting `users:write` to `operator` effectively gives all operators the ability to manage users — do this only if you have a flat team structure where all operators also manage access

**H1: Air-Gap Operation**
One-sentence intro: "This guide covers deploying Master of Puppets in a network-isolated environment where internet access is unavailable or restricted."

Opening context paragraph: "MoP is designed with air-gap deployments in mind. The package mirror sidecars and offline documentation plugins allow most of the stack to operate without outbound internet access. This guide documents the full setup, what is already offline-capable, what requires substitution, and how to verify isolation."

---

**H2: What Is Already Offline-Capable**

"The following components work without internet access after initial setup:"

| Component | How it achieves offline capability |
|-----------|-----------------------------------|
| Documentation site | MkDocs `privacy` + `offline` plugins pre-download all assets at build time — zero CDN requests at runtime |
| Foundry image builds | Python packages sourced from the configured `pypi_mirror_url`; APT packages from `apt_mirror_url` |
| Smelter CVE scanning | Operates against the mirrored package registry |
| Node execution | Scripts run locally inside containers — no outbound calls from the execution runtime itself |

---

**H2: Package Mirror Setup**

"The mirror sidecars must be configured and seeded before disconnecting from the internet."

**Step 1: Deploy the mirror sidecars**
The mirror sidecars (PyPI mirror and APT mirror) are included in `compose.server.yaml`. Confirm they are running:
```bash
docker compose -f puppeteer/compose.server.yaml ps
```

**Step 2: Configure mirror URLs**
Update the mirror configuration via the API:
```bash
curl -X PATCH \
  -H "Authorization: Bearer <TOKEN>" \
  -H "Content-Type: application/json" \
  https://<HOST>/admin/mirror-config \
  -d '{"pypi_mirror_url": "http://pypi-mirror:3141", "apt_mirror_url": "http://apt-mirror:3142"}'
```

This is logged as `mirror:config_updated` in the audit log.

**Step 3: Seed the mirrors**
Before disconnecting from the internet, ensure all required packages are present in the mirrors. Use the Smelter to pre-approve and mirror all ingredients your Foundry templates will need.

**Step 4: Add packages after initial setup**
"When a new package is needed after internet access is removed, an admin must upload the package file directly to the mirror:"
- PyPI: upload `.whl` or `.tar.gz` via **Admin** → **Smelter** → **Upload Package** — logs `smelter:package_uploaded`
- APT: upload `.deb` package to the APT mirror sidecar via its admin interface

---

**H2: Offline Build Validation**

"Before fully disconnecting, verify that Foundry builds succeed with the mirror configuration active and no internet access."

Procedure:
1. Configure mirror URLs (as above)
2. Block outbound internet access at the network/firewall level or use a Docker network policy
3. Trigger a Foundry build with `--no-cache` to force all dependencies to be re-downloaded from the mirror:
   - Navigate to **Templates** → select a template → click **Build**
4. Confirm the build succeeds — a failure at this stage means a dependency is not yet mirrored

Add `!!! tip`: "Run this validation before committing to air-gap mode. It is significantly easier to debug missing packages while internet access is still available."

---

**H2: Outbound Network Restrictions**

"Once mirrors are seeded and validated, apply network isolation:"

Recommended approach: firewall rules that allow only:
- Internal LAN traffic between orchestrator, nodes, and mirror sidecars
- Any required on-premises services (e.g., internal DNS, LDAP, SIEM)

Block all other outbound connections.

After applying restrictions, run the offline build validation procedure again to confirm no implicit internet dependency was missed.

"No additional steps are required — the documentation container is already offline-capable by default."

The MkDocs build (Phase 20) uses the `privacy` plugin to download all external assets (fonts, JavaScript) at build time and embed them in the Docker image. At runtime, the nginx container serves only local files.

**H2: What Still Requires Internet**

"The following components cannot be fully air-gapped without substitution. This list is exhaustive — if it is not here, it works offline."

| Component | Internet dependency | Suggested substitution |
|-----------|--------------------|-----------------------|
| Cloudflare Tunnel | Outbound to `*.cloudflare.com` for dashboard public access | On-prem reverse proxy: Traefik, nginx, or HAProxy with your own TLS certificate |
| Base image pulls (initial) | Docker Hub for `python:3.12-alpine`, `debian:12-slim`, `nginx:alpine` | Pre-pull all base images to a local registry (`docker pull` + `docker tag` + push to local registry) |
| PowerShell installation (DEBIAN blueprint) | Downloads `.deb` from `github.com/PowerShell/PowerShell/releases` | Mirror the specific PowerShell `.deb` release to a local artifact store and update the DEBIAN capability matrix injection recipe |
| Initial `pip install` in Foundry builds | If `pypi_mirror_url` is not configured, pip defaults to pypi.org | Configure mirror URL before any builds — see Package Mirror Setup above |

Add `!!! warning`: "This table reflects the current implementation. If you add custom blueprints or Foundry templates that install packages from external sources, those dependencies are not automatically mirrored — review each new template before deploying in air-gap mode."
