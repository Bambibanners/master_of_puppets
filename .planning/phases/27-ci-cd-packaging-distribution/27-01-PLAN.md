  <name>Task 1: Create CI workflow (ci.yml)</name>
  <files>.github/workflows/ci.yml</files>
  <action>
Create `.github/workflows/ci.yml` with four parallel jobs triggered on `push: branches: [main]` and `pull_request: branches: [main]`.

Job: `backend`
- `runs-on: ubuntu-latest`
- `strategy: fail-fast: false, matrix: python-version: ["3.10", "3.11", "3.12"]`
- Steps: `actions/checkout@v4`, `actions/setup-python@v5` (with `cache: pip`), `pip install -r puppeteer/requirements.txt`, then `cd puppeteer && pytest -v` with env vars:
  ```
  API_KEY: ci-dummy-key
  ENCRYPTION_KEY: AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=
  DATABASE_URL: sqlite+aiosqlite:///./test.db
  ```
- The env vars are CRITICAL — security.py calls sys.exit(1) at import if API_KEY is absent; without DATABASE_URL the aiosqlite import will fail on a clean runner.

Job: `frontend-lint`
- `runs-on: ubuntu-latest`
- Steps: `actions/checkout@v4`, `actions/setup-node@v4` (node-version: "20", cache: npm, cache-dependency-path: puppeteer/dashboard/package-lock.json), `cd puppeteer/dashboard && npm ci`, `cd puppeteer/dashboard && npm run lint`

Job: `frontend-test`
- `runs-on: ubuntu-latest`
- Same Node setup as frontend-lint (separate job for independent failure signals)
- `cd puppeteer/dashboard && npm ci`, `cd puppeteer/dashboard && npx vitest run`
- Use `npx vitest run` NOT `npm run test` — `npm run test` invokes `vitest` in watch mode which hangs in non-TTY CI environments

Job: `docker-validate`
- `runs-on: ubuntu-latest`
- Steps: `actions/checkout@v4`, `docker/setup-buildx-action@v3`, `docker/build-push-action@v6` with:
  - `context: puppeteer`
  - `file: puppeteer/Containerfile.server`
  - `push: false`
  - `platforms: linux/amd64`
  - No QEMU needed for amd64-only validation
  - No secrets needed — Containerfile.server has no build-time ARG secrets (all env vars are runtime-only)

Use workflow-level name: "CI"

<task type="auto">
  <name>Task 2: Create release workflow (release.yml)</name>
  <files>.github/workflows/release.yml</files>
  <action>
Create `.github/workflows/release.yml` with four jobs triggered on `push: tags: ['v*']`.

Workflow name: "Release"

Job: `build-python`
- `runs-on: ubuntu-latest`
- Steps: `actions/checkout@v4`, `actions/setup-python@v5` (python-version: "3.12"), `pip install build`, `python -m build`, then `actions/upload-artifact@v4` uploading path `dist/` with name `python-package-distributions`
- `python -m build` reads pyproject.toml — it will produce `dist/axiom_sdk-1.0.0a1-py3-none-any.whl` and `dist/axiom-sdk-1.0.0a1.tar.gz` (PEP 440 normalizes `1.0.0-alpha` to `1.0.0a1`)

Job: `publish-testpypi`
- `needs: [build-python]`
- `runs-on: ubuntu-latest`
- `environment: name: testpypi, url: https://test.pypi.org/p/axiom-sdk`
- `permissions: id-token: write`  ← scope ONLY to this job, not workflow level
- Steps: `actions/download-artifact@v4` (name: python-package-distributions, path: dist/), `pypa/gh-action-pypi-publish@release/v1` with `repository-url: https://test.pypi.org/legacy/`

Job: `publish-pypi`
- `needs: [publish-testpypi]`
- `runs-on: ubuntu-latest`
- `environment: name: pypi, url: https://pypi.org/p/axiom-sdk`
- `permissions: id-token: write`
- Steps: `actions/download-artifact@v4`, `pypa/gh-action-pypi-publish@release/v1` (no repository-url — defaults to real PyPI)

Job: `docker-release`
- `runs-on: ubuntu-latest`
- `permissions: contents: read, packages: write`  ← packages:write required for GHCR push
- Steps:
  1. `actions/checkout@v4`
  2. `docker/setup-qemu-action@v3` — required for arm64 emulation on amd64 runner
  3. `docker/setup-buildx-action@v3`
  4. `docker/login-action@v3` with registry: ghcr.io, username: ${{ github.repository_owner }}, password: ${{ secrets.GITHUB_TOKEN }}
  5. `docker/metadata-action@v6` (id: meta) with:
     - images: ghcr.io/axiom-laboratories/axiom
     - flavor: `latest=auto`
     - tags: `type=semver,pattern={{version}}` and `type=semver,pattern={{major}}.{{minor}}`
  6. `docker/build-push-action@v6` with:
     - context: puppeteer
     - file: puppeteer/Containerfile.server
     - platforms: linux/amd64,linux/arm64
     - push: true
     - tags: ${{ steps.meta.outputs.tags }}
     - labels: ${{ steps.meta.outputs.labels }}
     - cache-from: type=gha
     - cache-to: type=gha,mode=max

CRITICAL: Do NOT put `id-token: write` at the workflow level — this would grant OIDC token creation to the docker-release job which does not need it and creates a security surface.

The `docker-release` job runs independently of the Python publish jobs (no `needs:` dependency) — Docker and PyPI releases proceed in parallel after the tag push.
</action>
