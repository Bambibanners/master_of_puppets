## Unsatisfied Requirements — Phase 32 Gap

All 4 gaps stem from **Phase 32 not being started**. Phase goal:

> Operators can view complete execution history for any job or node in the dashboard, see retry state on in-progress and failed runs, inspect stdout/stderr output in a readable terminal view, and see environment tags on nodes.

All backend data APIs are complete and verified. Phase 32 is a frontend build on top of existing endpoints.

| Requirement | What's Missing |
|-------------|----------------|
| OUTPUT-03 | Dashboard view for stdout/stderr output on past executions |
| OUTPUT-04 | Execution history query panel — list all runs for a job definition or node |
| RETRY-03 | Dashboard shows "attempt N of M" badge on in-progress and failed runs |
| ENVTAG-03 | Nodes view shows env_tag per node with filter control |

**Backend APIs ready to consume:**
- `GET /api/executions/{id}/attestation` — attestation + stdout/stderr export
- `GET /api/dispatch/{job_guid}/status` — dispatch status with exit_code + attempt
- `ExecutionRecord` table: stdout/stderr/script_hash/attestation_verified/attempt_number columns all present
- `NodeResponse` exposes `env_tag` (Phase 31 verified)

## Cross-Phase Integration

Integration checker verified 14/15 connections. All 4 E2E flows complete.

### E2E Flow: Dispatch → Execute → Attest → Export (COMPLETE ✓)

```
POST /api/dispatch (env_tag)
  → create_job (env_tag persisted — fix from 31-04)
    → pull_work() (env_tag filter + retry policy in WorkResponse)
      → node execute_task() (script_hash + attestation bundle signed)
        → report_result() (stdout/stderr + attestation verified + attempt_number stored)
          → GET /api/executions/{id}/attestation (export for offline verification)
```

### Integration Issues

**Info-level only — no blocking failures:**
- `attestation_service.verify_bundle()` receives `node_id=job.node_id`. If `job.node_id` is `None` at result time (edge case), stores `ATTESTATION_FAILED` instead of `ATTESTATION_MISSING`. Happy path is unaffected.
