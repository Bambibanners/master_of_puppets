## Integration Findings

### Critical

**MISSING-01 — ExecutionRecordResponse missing Phase 29 fields**

`models.py` `ExecutionRecordResponse` (lines 594–605) exposes only pre-Phase-29 fields. `stdout`, `stderr`, `script_hash`, `hash_mismatch`, `attempt_number`, `job_run_id` are all written correctly to the DB by `job_service.report_result()` but are silently dropped in both `GET /api/executions` and `GET /api/executions/{id}`. Phase 32 has no API surface to consume this data.

**Fix required before Phase 32 planning:**
- Extend `ExecutionRecordResponse` in `models.py` with all Phase 29 fields
- Update both route handlers in `main.py` to pass `r.stdout`, `r.stderr`, `r.script_hash`, `r.hash_mismatch`, `r.attempt_number`, `r.job_run_id`

### High

**MISSING-02 — `history:read` not seeded for operator/viewer roles**

`OPERATOR_PERMS` and `VIEWER_PERMS` in `main.py` do not include `history:read`. Non-admin users receive HTTP 403 on both execution history endpoints.

**Fix required before Phase 32:**
- Add `history:read` to `OPERATOR_PERMS` in `main.py`
- Add migration SQL (ON CONFLICT DO NOTHING) for existing deployed DBs

### Low

**MISSING-03 — `docker-release` has no `needs:` gate on `build-python`**

`release.yml` `docker-release` job runs in parallel with `build-python`. A `build-python` failure does not block image publication, creating potential version skew. Workflow ran successfully end-to-end. Accept the risk or add `needs: [build-python]`.
